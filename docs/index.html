Give Claude a second opinion.

A local Model Context Protocol server that lets Claude step out of its own head — forwarding a summary to ChatGPT, Gemini, or Grok and bringing back an independent, critical reply. It closes the gap on information that exists in the world but isn't available to Claude, and surfaces genuinely different perspectives for consideration.

The purpose

Every model has blind spots. This is how Claude checks its own.

A single model answers from one training run and, at best, one web-search habit. That leaves real gaps — information that exists, but that this model can't see or didn't think to look for. A second opinion from a different vendor closes that gap.

Counter stale knowledge

An LLM often needs updated information but hasn't sought it — answering confidently from a training cutoff. A second model (especially with web search on) catches what's gone out of date since.

Close the information gap

The goal is to reach information that exists in the world but may not be available to Claude — whether it's behind a different model's training data or a different search index.

Different training data

ChatGPT, Gemini, and Grok were each trained on different corpora. Where one is thin, another may be rich — so asking across vendors widens what's actually known.

Different web search, varying results

Each provider's built-in search hits the live web differently and returns varying information. Comparing them turns one search into several independent ones.

The objective isn't to outvote Claude — it's to put differing, independent perspectives on the table so you can weigh them. Claude stays the driver; the second opinion is just another well-informed voice.

A perfect case for a second opinion: “best, right now” depends on current apps and features that shift faster than any training cutoff. Watch Claude answer, then fan the question out to three other models with web search — and reconcile what they each surface.

How it works

An MCP tool call, four steps deep

The server speaks MCP over stdio. Claude calls a tool; the server forwards your summary to one external model in a single, stateless turn and returns the reply as structured JSON.

1

You ask Claude

“Get a second opinion from Gemini on this plan.” Claude writes a self-contained summary of the context.

2

Claude calls the tool

second_opinion with a target_model and optional focus, system prompt, temperature, max tokens.

3

Server forwards it

Single-turn call to that vendor's API — with the configured reasoning effort and web search. No chat history leaves your machine.

4

Reply comes back

Structured JSON: the model's text, the real model id, token usage, and latency — or a clean, typed error.

Capabilities

Three flagships, one consistent interface

Each provider is called through its current recommended interface, in single-turn mode. Defaults track each vendor's flagship and can be overridden per provider.

TargetProvider & interfaceDefault modelSDK
chatgptOpenAI — Responses APIgpt-5.5openai ≥ 2.36
geminiGoogle — Interactions APIgemini-3.1-pro-previewgoogle-genai ≥ 1.55
grokxAI — OpenAI-compatible Responses APIgrok-4.3openai ≥ 2.36

Shape each call

focusNarrow the reviewer's attention to one aspect.
system_promptReplace the default reviewer persona for a different kind of feedback.
temperaturePass through a sampling temperature when the model honours it.
max_tokensCap the reply length for a quick verdict.

Set once per provider

reasoning_effortminimal · low · medium · high — how hard the model thinks before answering.
web_searchAttach the provider's built-in search tool — the key to fresh facts.
model overridePoint any target at a different model name, no code change.
timeout_secondsPer-request wall-clock bound (retries off, so it's exact).

Reachability checks

list_available_models reports which providers have a key and pass a low-cost probe — so Claude can tell you what's usable up front.

Graceful, typed errors

Failures never crash the call — you get success:false with a type like auth_failed, rate_limit, timeout, or content_blocked.

Keys stay local

API keys live only in a git-ignored config.json (or env vars). Nothing is logged by default; no conversation history is forwarded.

The default reviewer persona

You are acting as an external reviewer for a conversation the user is having with another AI assistant… Be direct, concrete, and critical. If you disagree with the framing or see a stronger alternative, say so explicitly. Do not pad with praise. State your confidence level when making factual claims.
Quickstart

Up and running in a few minutes

Requires Python 3.10+. Bring your own API key for any provider you want to use — a provider with no key is simply skipped.

1 · Install

# from the project root
py -3 -m venv .venv
.\.venv\Scripts\Activate.ps1
pip install -e .

2 · Configure

copy config.example.json config.json
notepad config.json   # add the keys you have

3 · Register with Claude

// MCP config (Claude Code / Desktop)
{
  "mcpServers": {
    "llm-second-opinion": {
      "command": "llm-second-opinion",
      "env": {
        "LLM_SECOND_OPINION_CONFIG":
          "C:\\src\\...\\config.json"
      }
    }
  }
}

4 · Ask Claude

# in any Claude conversation
"Get a second opinion from Gemini
 on the plan we just sketched —
 turn on web search."

Honest scope

What it deliberately doesn't do

This is a focused v1 for single-developer use. The boundaries are intentional.

Single-turn only — no conversation history is forwarded to the external model.
No response streaming — the full reply arrives at once.
No caching, cost tracking, or budget enforcement — watch your provider dashboards.
No server-side rate limiting — it relies on the upstream provider's limits.
Stdio transport only — no HTTP or remote-access mode in v1.
Reachability is a key/outage probe — it doesn't validate the exact model name.
